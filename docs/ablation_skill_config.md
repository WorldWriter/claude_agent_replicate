## 关键发现

### 1. Skill 配置整体有效，提升约 52%
- 平均得分从 0.2363 提升到 0.3587，相对提升 51.8%
- 成功率从 20% 提升到 28%

### 2. Skill 在结构化输出任务上优势最大
- **DI/DM 任务**: skill 的输出格式约束（JSON schema、CSV 精度规则等）帮助模型正确生成结果
- di-text-001, dm-csv-007 从 0 分提升到满分，说明无 skill 时模型输出格式不符评测要求

### 3. ML 聚类/分类任务受益明显
- ml-cluster-009: 0 → 0.6043
- ml-cluster-014: 0.044 → 1.0
- skill 中的 reflect-retrain cycle 和 threshold 策略可能发挥了作用

### 4. SA 任务 skill 出现负效应
- data-sa-001: 0.5 → 1.0（v3 更好）
- 可能 skill 的额外约束干扰了模型在统计分析任务上的自由发挥
- 样本量小（3 个任务），需更多数据验证

### 5. Visualization 仍是最大短板
- 即使有 skill，plot 类任务成功率极低（仅 plot-bar-004 成功）
- 无 skill 时 plot 全部失败
- 后处理步骤（Python → npy）是额外失败点

### 6. 存在少数 v3 反超的 case
- ml-regression-002: v3 满分 vs v2 零分 — 可能是随机性或 skill 约束导致的过度规范化
- data-sa-001: v3 满分 vs v2 半分 — SA skill 可能需要优化

## 消融实验 2: VerifyResult 工具的影响 (v4 vs v2)

### 实验设计

v4 在 v2 的基础上新增 `VerifyResult` 工具：
- **信息来源**: 从 `SubmitPlan` 的 `output_spec` 获取（agent 自己在规划阶段从 README/sample 分析得出），不读取 eval config，避免泄露
- **校验项**: 文件存在(MISSING)、列名匹配(COLUMNS)、NaN 检测(NAN)、行数对齐(ROW_COUNT)、JSON 有效性(JSON_ERROR)、空文件(EMPTY)
- **工作方式**: 纯 Python 校验，零额外 LLM 消耗；agent 完成任务前调用，FAIL 则自行修复

### 结果

| 指标 | v2 (skill) | v4 (skill+VerifyResult) | 差值 |
|------|-----------|------------------------|------|
| 平均得分 | **0.3587** | 0.3486 | -0.0101 |
| 成功率 | **28% (7/25)** | 24% (6/25) | -4pp |

### 分析

**VerifyResult 影响不显著**，v2 vs v4 差异（-1pp）在噪声范围内。

逐任务对比 v2 和 v4，大部分任务得分一致，差异来自模型随机性：
- **v4 新增成功**: plot-pie-008 (0→1.0)、ml-regression-002 (0→1.0)
- **v4 退化**: data-sa-026 (1.0→0，20轮未产出文件)、ml-multi-008 (1.0→0.73)、plot-bar-004 (1.0→0)

退化的 case 与 VerifyResult 无关 — data-sa-026 连文件都没生成就耗尽了 turns，根本没走到校验步骤。

### 原因推测

1. **VerifyResult 解决的是"格式对但内容错"的问题** — 而当前大部分失败是"根本没做出来"或"方法路径错误"，不是格式层面的问题
2. **25 个任务样本量太小** — 单个任务的随机波动（如 data-sa-026 从满分到零分）足以掩盖工具的微小收益
3. **skill 已经覆盖了格式指引** — base.md 中的通用规则（列名照抄 sample、输出文件名以规范为准）已经在 prompt 层面解决了大部分格式问题，VerifyResult 是二次保险

## 消融实验 3: PlanMode 的影响 (v5 vs v2)

### 实验设计

v5 关闭 PlanMode（`--plan-mode disabled`）：
- **System Prompt**: 使用 `base_noplan.md`（无阶段一/阶段二描述、无 SubmitPlan 工具说明）
- **工具列表**: 无 SubmitPlan，从第一轮起即可使用 WriteFile/RunCommand
- **工作流**: 直接读取文件→分析→执行，省去 Plan 阶段的 turns 消耗

### 结果

| 指标 | v2 (skill, plan on) | v5 (skill, plan off) | 差值 |
|------|---------------------|----------------------|------|
| 平均得分 | 0.3587 | **0.3865** | +0.0278 |
| 成功率 | 28% (7/25) | **32% (8/25)** | +4pp |

### 逐任务差异分析 (v2 → v5)

| 变化方向 | 任务 | v2 | v5 | 说明 |
|----------|------|-----|-----|------|
| **v5 提升** | ml-multi-003 | 0.00 | 0.41 | plan off 后直接执行反而做出了部分结果 |
| **v5 提升** | ml-regression-002 | 0.00 | 1.00 | 省去 plan 消耗，直接训练成功 |
| **v5 提升** | data-sa-001 | 0.50 | 1.00 | 自由度更高，完整分析 |
| **v5 退化** | ml-cluster-009 | 0.60 | 0.44 | plan 帮助了聚类策略选择 |
| **v5 退化** | ml-competition-003 | 0.05 | 0.00 | 微弱差异，噪声范围 |
| **v5 退化** | plot-bar-004 | 1.00 | 0.00 | 仅 v2 成功（随机性） |

### 分析

**关闭 PlanMode 略有提升**（+2.8pp / +7.8%），但差异在统计噪声范围内。

关键洞察：
1. **Plan 阶段消耗 turns** — 在 20 轮上限下，plan 阶段占用 2-4 轮（读取+分析+SubmitPlan+批准），留给实际执行的轮数更少
2. **简单任务无需规划** — DI/DM/SA 类任务结构清晰，直接执行效率更高
3. **复杂 ML 任务理论上受益于规划** — 但实际效果被 turns 限制抵消
4. **Plot 任务差异来自随机性** — plot-bar-004 仅 v2 成功，与 plan 无直接关系
5. **v5 在 ML 类上表现最好** — 0.4161 vs v2 的 0.3918，省下的 turns 用于更多训练迭代

## 结论

1. **Skill 配置是主要提升因素**（+12pp / +52%），其他变量影响远小于此
2. **PlanMode 影响不显著**（+2.8pp / +7.8%），关闭 plan 反而略好，说明在 20 轮限制下 plan 阶段的 turns 消耗可能得不偿失
3. **VerifyResult 工具影响不显著**（-1pp，噪声范围）
4. **SA 类任务** 的 skill 规则可能需要放宽或优化
5. **Plot 类任务** 需要更强的策略改进，当前 skill 帮助有限
6. 少数 case 的反转提示存在**随机性**，多次实验取平均会更可靠
7. **优化优先级**: Skill 策略改进 > 增加 max_turns > PlanMode 调优 > VerifyResult
