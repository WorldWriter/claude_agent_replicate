# 研究问题 2 Agent Loop 的实现机制

Claude Code 的 Agent Loop 是一个"思考 → 行动 → 观察"的无限循环，但它不是普通的 while 循环。它建立在一套 **async generator 流式架构**之上，把 LLM 流式输出、工具并发执行、上下文压缩、错误恢复等机制统一编织在同一个循环体里，同时又让调用方在第一个 token 到达时就能开始渲染 UI。

***

## 一、总览：三层结构

Agent Loop 的整体结构分为三层，从外到内依次是：

**入口层**（QueryEngine）— 持有会话状态，是每次对话的"状态盒"。每次用户发送消息，都通过它进入主循环。

**主循环**（queryLoop）— `while(true)` 结构，循环体是一次"LLM 调用 + 工具执行"的完整回合。它不是递归的，而是通过一个 **State 对象**在每轮之间传递状态，再用 `continue` 跳入下一轮。

**执行层**（工具编排）— 负责把 LLM 输出的工具调用列表，按照并发安全性分批执行，并把结果汇总返回给主循环。

```
用户输入
  │
  ▼
QueryEngine.submitMessage()         ← 入口层：构建 System Prompt，处理 slash 命令，持久化用户消息
  │
  ▼
queryLoop()  ←─────────────────┐
  │                            │
  ├─ [压缩预处理]               │
  ├─ LLM 流式调用               │   while(true)
  ├─ 工具并发执行                │
  ├─ 附件 & 记忆注入             │
  └─ 检查停止条件 ──────────────┘
        │
        └─ 无更多工具调用 → 完成
```

整个架构是一条 **async generator 链**：从内层的 LLM 流，到 queryLoop，再到 submitMessage，每一层都用 `yield` 向上透传消息。调用方（REPL、SDK、IDE）在收到第一个 token 时就能开始渲染，无需等待整个循环结束。

> **关于 `yield`**：可以把 generator 想象成一个流水线工人，`yield` 是他把零件递出手的动作。递出去之后他就站在那等，等对方接手并处理完，才继续干下一件事。加了 `async` 之后，这个工人在等零件加工（网络请求、工具执行）的时候不傻站着，而是去干别的事情，加工完了再回来接着递。因此，"后一个循环的执行"并不需要等前一个循环全部完成——只要没有数据依赖，就可以并行推进。这正是工具并发执行的理论基础：多个工具的执行可以同时进行，谁先完成就先把结果递出去。

> 参见 `src/QueryEngine.ts`：`submitMessage()` 入口；`src/query.ts`：`queryLoop()` 主循环

## 二、流程详解：一次完整轮次的五个环节

### 2.1 压缩预处理（LLM 调用前）

每轮 LLM 调用前，主循环会对消息历史执行一条**五级压缩管道**，从轻到重依次尝试，目标是把上下文控制在 context window 的安全范围内：

```
消息历史 messages
  │
  ├─ ① applyToolResultBudget    工具结果预算裁剪：对超长 tool_result 做截断/替换
  ├─ ② snipCompactIfNeeded      历史裁剪：移除最早的 N 条消息
  ├─ ③ microcompact              微压缩：缓存的轻量级压缩，压缩单条过长消息
  ├─ ④ applyCollapsesIfNeeded    上下文折叠：基于投影的细粒度消息摘要
  └─ ⑤ autocompact               全量压缩：用一次独立 LLM 调用将整段历史压缩为摘要
  │
  ▼
messagesForQuery（压缩后的消息，用于本轮 LLM 调用）
```

**设计意图**：压缩不是非此即彼的"有或没有"，而是一个梯度。轻量操作（①②③）尽可能保留原始上下文，只有在接近 context window 上限时才触发重量压缩（⑤）。压缩触发后，主循环**在当轮继续使用压缩后的消息**，而不是重新开始一轮——这样避免了空转。

自动压缩的触发条件：`context window 大小 - 预留输出 token - buffer` 的计算结果，超过阈值时触发。连续压缩失败 3 次后熔断，不再重试（`autoCompact.ts:67-70`）。

> 参见 `src/services/compact/` 目录下的各压缩模块；`src/query.ts:379-468` 压缩管道调用链

### 2.2 LLM 流式调用

压缩后的消息进入 API 调用阶段。这个阶段包含三个子步骤：请求构建、流式消费、异常处理。

**请求构建**：

1. **消息规范化**（`normalizeMessagesForAPI`）：修复 tool_use/tool_result 配对错误、去除 advisor blocks（无 beta header 时）、截断过多媒体（>100 项）
2. **System Prompt 组装**：5 层拼接——工具 schema（缓存）+ 核心指令 + 记忆/自纠正规则 + 动态上下文 + 格式规则。静态区和动态区之间有缓存分界线（`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`），减少 token 重复计算
3. **Cache breakpoint 注入**（`addCacheBreakpoints`）：在消息序列的关键位置插入缓存断点，最大化 prompt caching 命中率

**流式消费**：

API 返回 Server-Sent Events 流，主循环逐事件处理：

```
message_start        → 初始化 usage，捕获 research 字段
content_block_start  → 创建 text/tool_use/thinking block（空内容）
content_block_delta  → 增量累积文本/工具输入
message_delta        → 更新 usage 和 stop_reason
message_stop         → 将本消息 usage 累加到会话总量
```

每收到一个 `assistant` 消息，立即 `yield` 给上层（UI/SDK），实现逐 token 渲染。

**监控机制**：

| 机制 | 阈值 | 行为 |
|------|------|------|
| Idle Watchdog | 90 秒无 chunk | 中断流，报超时错误 |
| Stall Detection | 连续 chunk 间隔 >30 秒 | 记录日志（不中断） |
| TTFB 测量 | 首个 chunk 到达时 | 记录 Time-to-First-Byte 指标 |

**Withholding 模式 — 可恢复错误的暂扣机制**：

当 API 返回特定可恢复错误时，主循环**不立即暴露给 SDK 消费方**，而是先在内部尝试恢复：

| 错误类型 | 恢复方式 | 恢复失败后 |
|----------|---------|-----------|
| prompt_too_long (413) | 尝试 context collapse → reactive compact | yield error |
| max_output_tokens | 扩大到 64k → 注入"请继续"提示 | yield error |
| media_size_error | reactive compact 去除超大媒体 | yield error |

这意味着消费方看到的错误流是"已尽力恢复但仍失败"的最终结果，不会被中间的可恢复错误打断。

> 参见 `src/services/api/claude.ts`：`queryModel()` 流式调用；`src/query.ts:799-825` Withholding 逻辑

### 2.3 工具并发执行

LLM 响应中可能包含多个 `tool_use` block。Claude Code 提供两条执行路径：

**路径 A：StreamingToolExecutor（边流边执行）**

这是主路径。当 LLM 流还在输出时，每收到一个完整的 `tool_use` block，**立即启动该工具的执行**，不等流结束：

```
LLM token 流：
[text: "我来读取文件..."] → 
        ↓
[tool_use: ReadFile(a.ts)] →  流到这里时,立即启动读 a.ts
        ↓
[tool_use: ReadFile(b.ts)] →  流到这里时,立即启动读 b.ts（并行）
        ↓
[text: "..."]
```

在流结束后，主循环调用 `getRemainingResults()` 收集剩余工具结果，此时大多数工具可能早已执行完毕。

**路径 B：runTools（批量执行）**

备选路径，在 StreamingToolExecutor 不可用时使用。先收集所有 tool_use block，再按并发安全性分批执行。

**并发策略（两条路径共享）**：

并非所有工具都可以并行。每个工具有 `isConcurrencySafe` 属性，分批规则如下：

```
partitionToolCalls() 分批规则：
  ├─ 连续的"安全工具"合并为一批 → 并发执行（如多个 ReadFile、Grep）
  └─ "非安全工具"单独成一批 → 串行执行（如 BashTool 写操作、EditFile）

执行顺序：批次之间串行，批次内部并发
```

**Bash 失败级联取消**：当一个 Bash 工具执行失败时，会通过 `siblingAbortController.abort()` 取消同批次中其他正在执行的工具。原因是 Bash 命令通常有隐式依赖链（如 `mkdir` 失败后后续命令无意义）。但 Read、WebFetch 等独立工具的失败**不会**触发级联取消。

> 参见 `src/services/tools/StreamingToolExecutor.ts`；`src/services/tools/toolOrchestration.ts`：`partitionToolCalls()`

### 2.4 附件 & 记忆注入（工具执行后、下一轮 LLM 调用前）

工具执行结束后，主循环在构建下一轮消息前，会注入多种系统信息。这些信息以 `user` 角色的 `<system-reminder>` 消息追加在消息历史末尾。

**记忆预取（Memory Prefetch）**：

记忆的检索不是在这个阶段才开始的——它在**本轮 LLM 调用启动的同时**就已经并行启动了：

```
时间线：
  LLM 流式调用开始 ──────────────────────── LLM 流结束
       │                                       │
       └─ 同时启动 memory prefetch              │
                │                               │
                └─ prefetch 完成（通常更快）     │
                                                │
                                      工具执行 ─┘
                                                │
                                      消费 prefetch 结果 ← 此时注入
```

`findRelevantMemories()` 用一次 Sonnet 调用，从内存文件中挑选最多 5 个与当前查询语义相关的记忆（每文件 ≤4KB），会话累计上限 60KB。已被模型 Read/Write/Edit 过的文件会被过滤，避免重复注入。

**Skill Discovery Prefetch**：与记忆预取类似的并行机制，在 LLM 流式调用期间检索与当前任务匹配的 Skill 推荐。

**队列命令消费**：用户在模型生成期间发送的新消息、task-notification 等排队命令，在此阶段被消费并注入。

**Tool Use Summary**：工具执行期间，会异步调用 Haiku 生成工具行为摘要（约 1 秒，远快于一轮 5-30 秒的工具执行）。摘要在此阶段 yield 给上层，用于 UI 展示。子 Agent 中不生成。

**缓存友好性**：附件**不改动 System Prompt**，因此不破坏静态区的全局缓存。它们作为 `user` 消息追加在消息历史末尾，只影响 prefix cache 的最末段。

> 参见 `src/utils/attachments.ts`：`getAttachmentMessages()`；`src/memdir/findRelevantMemories.ts`

### 2.5 停止条件检查

#### 核心停止条件

循环的核心停止条件只有一个：

> **模型本轮响应中没有任何 `tool_use` block → `needsFollowUp = false` → 进入退出流程**

退出流程依次检查：

```
needsFollowUp = false
   ↓
① API 错误检查（prompt_too_long、max_output_tokens 等可恢复错误的处理）
② handleStopHooks()  → 运行用户配置的外部脚本
③ token budget 检查  → 未超出预算
④ return { reason: 'completed' }  → 循环结束
```

#### 绝大多数场景：prompt 软约束驱动的任务验证

**普通用户从不配置 Stop Hook，Claude Code 依然能自己跑测试、验证结果。** 这是通过 prompt 软约束实现的——System Prompt 和 CLAUDE.md 里的指令影响模型的自主行为：

```
CLAUDE.md（或 System Prompt）里写：
  "run lint/typecheck/test before marking done"
            ↓
模型推理：我刚改了代码，应该运行测试
            ↓
tool_use: Bash("npm test")  ← 模型自主决定调用
            ↓
tool_result: 测试失败
            ↓
模型推理：需要修复
            ↓
tool_use: Edit(...)  →  tool_use: Bash("npm test")
            ↓
tool_result: 所有测试通过
            ↓
模型推理：任务完成，不再调用工具
            ↓
needsFollowUp = false → return { reason: 'completed' }
```

这整个过程由**模型推理**驱动，没有任何代码在判断"测试是否通过"。因此它有固有局限：模型可能判断失误、遗忘指令、或跳过检查。

#### Stop Hook：用户可选的外部硬约束

Stop Hook 是面向高级场景的**可选扩展**，默认不存在。用户手动写入 `~/.claude/settings.json` 或 `.claude/settings.json`：

```json
{
  "hooks": {
    "Stop": [{ "hooks": [{ "type": "command", "command": "npm test" }] }]
  }
}
```

**脚本通信协议**（通过 stdout 返回 JSON）：

```
{ "continue": true }                            → 通过，循环正常退出
{ "continue": false, "stopReason": "测试失败" } → 阻止退出，stopReason 以
                                                   user 消息注入下一轮，
                                                   强制模型修复后再次触发检查
非零退出码 + stderr                             → 非阻断错误，记录但不阻止退出
```

Stop Hook 还支持 `preventContinuation` 模式：直接阻止循环继续，不注入消息让模型重试，而是立刻终止。

#### 三种约束层次对比

`/init` 命令里，Claude Code 对自己的约束体系有明确的三层分类（`src/commands/init.ts`）：

| 约束层次 | 机制 | 可靠性 | 典型用途 |
|---------|------|--------|---------|
| **Hook**（硬约束） | 确定性 shell 脚本，Claude 不能跳过 | 最高 | 格式化、lint、跑指定测试 |
| **Skill**（按需调用） | 用户或 Claude 主动触发 `/skill-name` | 中 | 完整验证流程、部署、发布 |
| **CLAUDE.md note**（软约束） | 影响模型行为但不强制执行 | 较低 | "先规划再编码"、"解释权衡" |

#### 异常退出

| 情况 | 退出原因 |
|------|----------|
| 用户中断（streaming 阶段） | `aborted_streaming` |
| 用户中断（tool 执行阶段） | `aborted_tools` |
| Stop Hook 阻止 | `hook_stopped` / `stop_hook_prevented` |
| 达到 maxTurns 上限 | `max_turns` |
| 上下文过长且无法压缩恢复 | `prompt_too_long` |
| 上下文达到 blocking limit | `blocking_limit` |
| 模型报错 | `model_error` |

> 参见 `src/query/stopHooks.ts`；`src/types/hooks.ts`

## 三、亮点设计

### 3.1 State 对象 — 为什么不用递归

主循环最直觉的实现是递归：每轮结束时调用自身，传入更新后的消息列表。但 Claude Code 选择了 **while(true) + State 对象**。

**State 对象**在每轮循环顶部被解构读取，在每轮底部被重建并赋值给 `state`，然后 `continue` 跳回顶部：

```
State = {
  messages          ← 完整消息历史（含本轮新增的 tool_result）
  toolUseContext    ← 工具权限、模型选择、abort 信号等运行时上下文
  turnCount         ← 当前轮次（用于 maxTurns 检查）
  autoCompactTracking  ← 压缩状态追踪
  maxOutputTokensRecoveryCount  ← 错误恢复计数（最多重试 3 次）
  hasAttemptedReactiveCompact   ← 防止压缩死循环的标志
  pendingToolUseSummary         ← 上一轮异步生成的工具摘要
  transition        ← 本轮因何进入（调试与分析用）
}
```

**设计意图**：递归在极长任务中会栈溢出，且每轮都产生额外的 generator 帧。而用 State 对象存储中间状态，循环本身只有一个帧，`transition` 字段还额外记录了"本轮为什么 continue"——是正常的下一轮、还是 max_output_tokens 恢复、还是 reactive compact 触发——测试可以直接断言，不需要检查消息内容。

> 参见 `src/query.ts`：State 类型定义（约第 204-217 行）和 `while(true)` 主循环（第 307 行起）

### 3.2 错误恢复 — 循环内的自愈

Agent Loop 的一个核心设计是：**错误不应该直接终止循环，而应该尝试在循环内部恢复**。主要有三种恢复路径：

| 错误类型 | 恢复策略 | 最大重试 |
|----------|----------|----------|
| **输出 token 超限** | 先尝试扩大 max_output_tokens 到 64k；如仍超限，注入"请继续"提示让模型接续输出 | 3 次 |
| **上下文过长（413）** | 先尝试 context collapse 释放空间；若不够，触发 reactive compact 全量压缩 | 各 1 次 |
| **模型过载触发 fallback** | 切换到备用模型，丢弃本轮已有的流式结果，重新发起请求 | 1 次 |

所有恢复路径都通过 `state = { ..., transition: { reason: '...' } }; continue` 实现，和正常下一轮的推进机制完全一样。这保证了恢复逻辑不是特殊路径，而是主循环的自然组成部分。

可恢复错误在暴露给消费方之前会被**暂扣**（Withholding），详见 2.2 节。

### 3.3 Plan Mode — 工具驱动的模式切换

Plan Mode 不是 Agent Loop 的一个分支路径，而是通过**替换工具集**来改变模型的行为边界，循环本身的结构完全不变。

**进入 Plan Mode 的过程：**

模型调用 `EnterPlanMode` 工具（一个普通的 tool_use）。工具执行时，系统把权限模式从 `default/auto` 切换到 `plan`，并把进入前的模式保存在 `prePlanMode` 字段中以便退出时恢复。工具的 `tool_result` 里写入一段 prompt，告诉模型："你现在处于只读的规划阶段，专注探索代码库和设计方案，不要写文件。"

**Plan Mode 期间，循环有什么不同：**

Agent Loop 本身毫无变化。区别在于**模型可以调用的工具集**发生了根本改变：

```
默认模式工具集：Read, Bash, Edit, Write, Grep, Agent, ...

Plan Mode 工具集（物理过滤掉写工具）：
  ✓ 保留：Read, Grep, Glob, Bash（仅只读命令）, ExitPlanMode
  ✗ 移除：FileEdit, FileWrite, NotebookEdit
  ✗ 移除：Agent（防止嵌套派生子 Agent）
```

Bash 工具依然存在，但 System Prompt 中有明确指令禁止写操作（`mkdir, touch, git commit...`），是 prompt-level 的约束；而其他写工具被物理移除，是 schema-level 的约束。两层防线共同保证规划阶段的只读性。

**退出 Plan Mode 的过程：**

模型把设计方案写入 `PLAN.md`，然后调用 `ExitPlanMode` 工具。这个工具需要用户手动确认（`requiresUserInteraction: true`），确认后系统把权限模式从 `plan` 恢复到 `prePlanMode`（原来的 `default` 或 `auto`），并把完整的 plan 文本通过 `tool_result` 返回给模型，模型就此进入实现阶段。

在多 Agent（Teammate）场景中，退出逻辑有所不同：Teammate 不需要本地用户确认，而是把 plan 通过信箱（mailbox）发送给 Team Leader，Leader 审批后才能继续实现。

**Plan Mode 完整消息序列示例：**

```
assistant:  tool_use: EnterPlanMode()
                    ↓
user:       tool_result: "Entered plan mode. DO NOT write or edit any files..."
                    ↓
user:       <system-reminder>plan_mode (full): 完整规划阶段规则</system-reminder>  ← plan_mode attachment（首次立即注入）
                    ↓
... plan mode 期间的多轮对话（每 5 个人类轮次注入一次 plan_mode attachment 提醒） ...
                    ↓
assistant:  tool_use: ExitPlanMode({ plan: "..." })
                    ↓
user:       tool_result: "User has approved your plan. You can now start coding...  ← ExitPlanMode 工具返回，含完整 plan 文本
             ## Approved Plan: ..."
                    ↓
user:       <system-reminder>## Exited Plan Mode                                    ← plan_mode_exit attachment（一次性注入）
             You have exited plan mode. You can now make edits...</system-reminder>
```

进入和退出各有**两条通道**配合：工具的 `tool_result` 传递具体指令和内容；附件系统的 `<system-reminder>` 周期性强化模式状态。退出时 `tool_result` 负责传递 plan 内容和"开始编码"的指令，`attachment` 负责告知模型"写工具现在可用了"。

**设计意图**：Plan Mode 是一个纯工具层的开关，不需要修改 Agent Loop 的任何逻辑。同一套循环基础设施，只靠替换工具集就能切换模型的"工作模式"——这体现了工具驱动架构的灵活性。

> 参见 `src/tools/AgentTool/built-in/planAgent.ts`、`src/tools/EnterPlanModeTool/`、`src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts`

### 3.4 Prompt Caching 优化

Agent Loop 对 prompt caching 做了多层优化，减少重复 token 计算的开销：

**System Prompt 静态/动态分区**：System Prompt 由 5 层组成，其中工具 schema 和核心指令是**静态**的（跨轮不变），动态上下文（日期、git 状态等）每轮可能变化。两者之间有一个 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 分界线，API 可以只重新计算动态区。

**Cache breakpoint 注入**：`addCacheBreakpoints()` 在消息序列的关键位置插入缓存标记，帮助 API 识别可复用的前缀段。

**附件不破坏缓存**：附件作为 `user` 消息追加在历史末尾，不修改 System Prompt 和已有消息——只影响 prefix cache 的最末段（通常已没有可用缓存）。

> 参见 `src/services/api/claude.ts`：`addCacheBreakpoints()`；`src/QueryEngine.ts:284-325`：System Prompt 组装

## 四、附录

### 附录 A：消息角色系统 — user / assistant / system

Anthropic API 的请求结构里有两个完全不同层次的字段：

```json
{
  "system": "你是一个编程助手...",   ← 顶层参数，不在 messages 里，不是 role
  "messages": [
    { "role": "user",      "content": ... },
    { "role": "assistant", "content": ... }
  ]
}
```

`system` 是独立的顶层参数，**messages 数组里只有两种 role：`user` 和 `assistant`**。没有 `system` role，也没有 `tool` role。

**`content` 字段的数据结构：ContentBlock 数组**

每条消息的 `content` 可以是字符串（纯文本的简写），也可以是**对象数组**，每个对象叫 ContentBlock，必须包含 `type` 字段：

```json
{ "role": "assistant", "content": [
    { "type": "text",     "text": "我来读取文件..." },
    { "type": "tool_use", "id": "call_abc", "name": "ReadFile", "input": { "file_path": "foo.ts" } }
]}

{ "role": "user", "content": [
    { "type": "tool_result", "tool_use_id": "call_abc", "content": "export function..." },
    { "type": "text",        "text": "<system-reminder>今天是 2026-04-02</system-reminder>" }
]}
```

常见 ContentBlock 类型：

| type | 出现在哪种 role | 含义 |
|------|--------------|------|
| `text` | user / assistant | 普通文本，system-reminder 也是这里 |
| `tool_use` | assistant | 模型发起的工具调用（含工具名和入参） |
| `tool_result` | user | 工具执行的返回结果（与 `tool_use` 的 id 对应） |
| `image` | user | 图片内容块 |
| `thinking` | assistant | 扩展思考内容（extended thinking 模式） |

**"user" role 是个"大杂烩"**：真实用户输入、工具结果、系统注入上下文，三者都以 `user` role 发送，区分靠 content block 的 type 和文本内容里的 `<system-reminder>` 标签（Claude Code 用 `isMeta: true` 在内部标记后两类，避免展示给用户界面）：

| user 消息来源 | content 内容 | isMeta |
|--------------|-------------|--------|
| 真实用户输入 | `type: "text"` | false（显示给用户） |
| 工具执行结果 | `type: "tool_result"` | false（显示为工具输出） |
| 系统注入（附件） | `type: "text"` + `<system-reminder>` | true（界面不显示） |

**设计原因**：对话模型在训练时假设对话是 user/assistant 交替——这是最简洁的二元结构。额外的信息渠道（工具、系统）被映射进已有的 user role 里，避免引入新 role 带来的训练分布问题。`<system-reminder>` 标签在语义上帮助模型区分"用户说的"和"系统提示"。

### 附录 B：附件注入系统 — 循环的"神经系统"

**附件（Attachment）** 是 Claude Code 在每轮工具执行结束后，通过 `getAttachmentMessages()` 往消息历史追加的系统注入消息。它们统一包裹在 `<system-reminder>` 里，以 `user` 角色发送，模型可读但界面不显示。

这套系统有 **50+ 种附件类型**，可以理解为：**循环缺少自我感知能力，附件系统就是它的神经信号——把"当前环境发生了什么变化"持续告诉模型**。

#### 触发规则的性质：全部是固定代码逻辑，不是 LLM 推理

**所有附件的触发判断都是确定性的代码规则**，在 `getAttachments()` 里并行计算，每次调用都会独立决策是否注入。规则的输入只有两种：

- **扫描消息历史**：倒序遍历 messages 数组，计数器式地找"上次某类附件/工具调用/人类消息"的位置
- **读取进程状态**：读取全局标志（如 `needsPlanModeExitAttachment`）、settings、AppState

没有模糊判断，也没有 LLM 参与决策。

#### 五类附件的精确触发规则

**① 模式状态通知**

| 附件类型 | 精确触发规则 |
|----------|------------|
| `plan_mode` | **检查**：当前 `permissionContext.mode === 'plan'`。**节流**：倒序扫描消息历史，统计距上次 `plan_mode` attachment 之间的"真实人类轮次"数（排除 meta 消息和 tool_result 消息）。*首次进入 Plan Mode 立即注入；此后每满 5 个人类轮次注入一次*。**内容**：第 1、6、11… 次为 full（完整规则），其余为 sparse（精简版） |
| `plan_mode_exit` | **检查**：全局标志 `needsPlanModeExitAttachment === true`（由 ExitPlanMode 工具设置）。*一次性注入，注入后立即清除标志* |
| `auto_mode` | 同 plan_mode 的节流逻辑，但检查 `TRANSCRIPT_CLASSIFIER` feature flag 和 Auto Mode 状态 |
| `auto_mode_exit` | 同 plan_mode_exit，检查 `needsAutoModeExitAttachment` 标志 |

**② 记忆注入**

| 附件类型 | 精确触发规则 |
|----------|------------|
| `relevant_memories` | 每轮循环**开始时**（与 LLM 调用并行）预取记忆，`getAttachments` 时消费结果。只注入语义相关的文件，全局上限：累计注入超过 60KB 后本会话停止预取。每文件 ≤4KB，每次最多 5 个文件 |
| `nested_memory` | 扫描本地 CLAUDE.md，若发现 `@import` 或嵌套引用的新文件，读取并注入。内容变化时重新注入 |

**③ 工具集变化增量**

这类附件使用**增量检测（diff）**，不是每次都全量注入：

倒序扫描消息历史，找到上一次同类型的 delta attachment，计算当前工具集与"上次已通知状态"的差集（addedNames / removedNames），只有差集非空时才注入。规则纯数组比较，无阈值。

**④ 环境与上下文变化**

| 附件类型 | 精确触发规则 |
|----------|------------|
| `queued_command` | 进程内有排队命令（用户中途发新消息、task-notification 到达），*无条件注入*，一次消费一批 |
| `diagnostics` | LSP 诊断注册表里有**新增**诊断（与上次注入后的状态做比较），有新内容才注入 |
| `date_change` | 当前本地日期（YYYY-MM-DD）与上次记录的 `lastEmittedDate` 不同，注入后更新记录 |
| `task_status` | 后台 Agent 任务状态发生变化时，由任务执行框架主动推入队列 |

**⑤ 行为提醒（节流逻辑最典型，以 `todo_reminder` 为例）**

`todo_reminder` 触发条件是**两个独立计数器同时满足**：

```
条件 A: 距上次 TodoWrite 工具调用 ≥ 10 个 assistant 轮次
         （倒序扫描 assistant 消息，找 tool_use.name === 'TodoWrite'）

条件 B: 距上次 todo_reminder attachment ≥ 10 个 assistant 轮次
         （倒序扫描 attachment，找 type === 'todo_reminder'）

A && B 同时为真 → 注入提醒，附带当前 todo list 完整内容
```

**关键**：条件 A 检查的是"最近是否用了 TodoWrite"，因此如果模型在最近 10 轮内调用过 TodoWrite，**不会触发**提醒。提醒的目的是"发现模型忘记了"，而不是"强制每 10 轮提醒"。

> **todo list 的可见性**：todo 内容**不是每轮都注入**。AppState 里的 `todos` 字段是 source of truth（由每次 `TodoWrite` 调用即时更新），附件只在触发条件满足时才把它读出来附进 context。中间的 8~9 轮里，模型靠自己在消息历史里追溯上次 `TodoWrite` 的 tool_result 来"记住"进度——提醒触发时相当于一次强制同步，把最新的 todo 状态显式贴回 context。

`output_style` 规则则更简单：读取 `settings.outputStyle` 字段，非 `'default'` 时就注入，**每轮都会触发**（不做节流），因为输出风格需要持续生效。

### 附录 C：全景图 — 一次完整的对话轮次

以"帮我读取 foo.ts 并总结"为例，展示从用户输入到最终回复的完整过程：

```
┌─ 用户发送消息 ──────────────────────────────────────────────────────┐
│                                                                    │
│  "帮我读取 foo.ts 并总结"                                           │
│                                                                    │
│  → 消息写入 transcript（在进入循环前，保证进程被杀也能 resume）        │
└────────────────────────────────────────────────────────────────────┘
                │
                ▼
┌─ queryLoop 第 1 轮 ─────────────────────────────────────────────────┐
│                                                                    │
│  [压缩预处理]                                                       │
│  → 检查上下文长度，决定是否需要压缩（首轮通常不触发）                  │
│                                                                    │
│  [LLM 流式调用]  ──────── 同时启动 memory prefetch ──────           │
│  → 模型开始输出：先是思考文本"我来读取..."                            │
│  → 接着输出 tool_use: ReadFile(foo.ts)                              │
│  → token 流到 tool_use 时，立即在后台启动文件读取，LLM 流继续        │
│  → LLM 流结束，needsFollowUp = true（有工具调用）                   │
│                                                                    │
│  [工具执行]                                                         │
│  → ReadFile 执行完成（可能在 LLM 流结束前就已完成）                  │
│  → 返回 tool_result: "文件内容：export function..."                 │
│                                                                    │
│  [附件 & 记忆注入]                                                  │
│  → 消费 memory prefetch 结果（已在 LLM 调用时并行完成）               │
│  → 检查队列命令、文件变更通知等                                      │
│                                                                    │
│  [推进下一轮]                                                       │
│  → messages 更新为 [原消息..., assistant 回复, tool_result, 附件]   │
│  → turnCount = 2，state = next，continue                           │
└────────────────────────────────────────────────────────────────────┘
                │
                ▼
┌─ queryLoop 第 2 轮 ─────────────────────────────────────────────────┐
│                                                                    │
│  [压缩预处理]                                                       │
│  → 检查上下文长度（两轮后通常还在安全范围内，不触发）                  │
│                                                                    │
│  [LLM 流式调用]                                                     │
│  → 模型看到文件内容后，输出最终总结文本                               │
│  → 本轮没有 tool_use block，needsFollowUp = false                  │
│                                                                    │
│  [停止条件检查]                                                     │
│  → 无 tool_use → 检查 stop hooks → 通过                            │
│  → 检查 token budget → 未超出                                      │
│  → return { reason: 'completed' }                                 │
└────────────────────────────────────────────────────────────────────┘
                │
                ▼
         用户看到最终回复

── 上下文压缩场景（长对话后触发）──────────────────────────────────────

  假设第 50 轮循环时，上下文接近 context window 上限：

  [压缩预处理]
  → autocompact 触发：用一次独立的 LLM 调用，把历史压缩为摘要
  → messagesForQuery 替换为摘要消息，当轮继续（不重启循环）
  → LLM 用压缩后的上下文继续第 50 轮的工作，不感知压缩的发生
```
