# 研究问题4 — Tool 系统深度解析

Claude Code 的工具系统是整个 CLI 的执行引擎。模型通过工具完成几乎所有实际操作——读写文件、执行命令、搜索代码、调用外部服务。工具不只是一组函数的集合，它是一套**能力契约**：每个工具向系统声明自己能做什么、不能做什么、什么情况下需要问人，系统据此做出调度、安全和渲染决策。

***

## 总览

```
┌──────────────────────────────────────────────────────┐
│                  Tool Pool（工具池）                   │
│                                                      │
│   内置工具（~45）          MCP 外部工具（动态注入）     │
│   ────────────────         ───────────────────────    │
│   执行原语                   任意语言实现               │
│   TypeScript 实现            通过适配器统一接口         │
└──────────────────────────────────────────────────────┘
            │
            │  组装时：deny rules 过滤 + 排序稳定化
            ▼
┌──────────────────────────────────────────────────────┐
│               Permission System（权限系统）            │
│   Hooks（预审批）→ Classifier（自动判断）→ User（人工） │
└──────────────────────────────────────────────────────┘
            │
            │  执行时：部分工具启动子模型调用
            ▼
┌──────────────────────────────────────────────────────┐
│                   Tool Execution                     │
│   同步结果 / 子模型结果 / 超限输出持久化到磁盘          │
└──────────────────────────────────────────────────────┘
```

> 参见 `src/Tool.ts`（接口定义）、`src/tools.ts`（注册与组装）、`src/hooks/toolPermission/`（权限系统）

## 一、内置工具：能力地图

内置工具约 45 个，但数量不是重点——分组方式才揭示设计意图。工具按**能力域**而非功能相似性组织，每一组代表 Claude Code 开放给用户的一项能力边界。

### 核心执行工具（始终可用）

| 工具 | 功能 |
|------|------|
| `Bash` | 执行 shell 命令——最通用的执行原语 |
| `Read` / `Edit` / `Write` | 文件三件套：读取、精确替换、覆盖写入 |
| `Glob` / `Grep` | 文件搜索与内容搜索（底层用 ripgrep）|
| `WebFetch` / `WebSearch` | 网络访问的两种形态：直接抓取 vs 引擎检索 |
| `Agent` | 启动子代理——工具调用工具的递归能力 |
| `Skill` | 执行用户定义的工作流（/slash-command 的背后）|
| `AskUserQuestion` | 主动向用户提问，是"自主执行"的对立面 |
| `ToolSearch` | 按需加载工具 schema（见第九节）|

### 结构性工具组

**Task 管理组**（`Create / Get / Update / List / Stop / Output`）：这组工具将模型的"思考步骤"外化为可跟踪的任务对象，支持长任务的中断、恢复和进度可视化。它们通过运行时检查 `isTodoV2Enabled()` 控制开关，说明这是一个可独立演进的子系统，而非核心依赖。

**Plan Mode 工具组**（`EnterPlanMode / ExitPlanMode / EnterWorktree / ExitWorktree`）：成对出现，代表"进入受限执行环境"的元操作。Plan Mode 下工具调用被拦截，模型只能规划不能执行；Worktree 模式在隔离的 git 分支中操作，避免污染工作区。工具的存在本身就是一种状态转换机制。

### Feature Flag 门控工具

这类工具揭示了 Claude Code 的**产品路线图**——被关闭的功能不是"没有"，而是"还没开放"：

| 能力方向 | 代表工具 | 说明 |
|---------|---------|------|
| 定时自动化 | `Cron*`、`RemoteTrigger` | 代理的主动调度能力 |
| 主动感知 | `Sleep`、`Monitor` | 代理等待条件、监控环境变化 |
| 浏览器操作 | `WebBrowser` | 超越 fetch 的完整网页交互 |
| 多代理协作 | `SendMessage`、`Team*` | 代理间通信与团队编排 |
| 工作流脚本 | `Workflow` | 结构化多步任务定义 |

> 参见 `src/tools.ts:25-135` 的条件注册逻辑；`src/services/analytics/` 的运行时 feature gate 检查

## 二、注册机制：工具可用性的多维控制

工具系统需要回答一个核心问题：**在任意一次对话中，模型应该能看到哪些工具？** 这个问题看似简单，实际上涉及多个相互独立的维度——产品功能开关、部署环境差异、用户权限配置、运行时上下文。

Claude Code 的选择不是用一个大型条件判断处理所有情况，而是构建了一条**渐进式精化管道**，每一层只负责自己关心的维度。

### 2.1 能力与可见性分离

工具的"存在"和"可见"是两件独立的事。这种分离让不同角色可以独立控制工具的可见范围，且互不干扰：

- **Anthropic 工程师**：通过编译期 feature flag 决定哪些功能打包进产品
- **企业管理员**：通过策略配置禁用工具，且**模型根本不知道这些工具的存在**
- **用户**：通过 `--allowedTools` 或 settings.json 按需调整工具集
- **工具自身**：通过 `isEnabled()` 声明在当前环境是否适用

每一层都只做"减法"——从全集筛选出当前场景的合理子集。

### 2.2 三个时间维度的控制

工具可用性的决策发生在三个不同时间点：

**编译期** — 产品形态决定。实验性工具通过 Bun 的 `feature()` 常量折叠在编译时决定取舍，关闭的工具代码本身就不存在于最终产物中。这同时是安全措施和包体积优化。

**启动期** — 部署环境决定。平台相关工具（Windows 的 PowerShell）、内部工具（Anthropic 专属的 REPL）在进程启动时一次性判断，之后不再变化。

**请求期** — 上下文决定。每次组装工具池时，deny rules、运行模式（REPL/Plan/Simple）、`isEnabled()` 共同决定最终工具集。同一进程的不同会话可能看到不同工具集。

### 2.3 "模型不知道"作为安全边界

被 deny 规则封锁的工具，在工具池**组装阶段**就从列表中移除——而不是等到调用时再拒绝。模型的 system prompt 中根本不包含这些工具的 schema，模型没有机会尝试调用它们。

相比"调用时报错"，"模型不知道"是更彻底的安全边界：它消除了模型绕过限制的动机，也避免了"被拒绝后反复重试"的行为模式。

### 2.4 管道全景

```
全集（编译期确定的所有工具）
    │
    │  编译期 feature flags → 关闭的功能代码不存在
    ▼
环境集（当前部署/平台可用的工具）
    │
    │  deny rules → 从 system prompt 中彻底移除
    │  isEnabled() → 工具自我声明当前是否适用
    ▼
有效集（当前配置允许的工具）
    │
    │  合并 MCP 外部工具（内置优先，去重）
    │  固定排序 → prompt cache 稳定性
    ▼
工具池（模型看到的最终工具集）
```

> 参见 `src/tools.ts` 的 `getAllBaseTools()` → `getTools()` → `assembleToolPool()` 调用链

## 三、Tool 接口：自描述的能力契约

每个工具不只是一个函数——它是一份**自描述的契约**，向系统声明四类信息：执行逻辑、调度约束、权限边界、渲染表现。系统读取这些声明后，无需检查工具内部实现就能做出正确决策。

```
┌────────────────────────────────────────┐
│              Tool 接口                  │
│                                        │
│  [执行]  call()                         │  ← 做什么
│                                        │
│  [调度]  isReadOnly()                   │
│          isDestructive()               │  ← 调度器据此决定
│          isConcurrencySafe()           │    并发/串行/警告
│          requiresUserInteraction()     │
│                                        │
│  [权限]  checkPermissions()            │  ← 权限系统据此
│          validateInput()              │    决定是否执行
│                                        │
│  [渲染]  renderToolUseMessage()        │
│          renderToolUseProgressMessage()│  ← UI 据此渲染
│          renderToolUseRejectedMessage()│    每个状态的样子
└────────────────────────────────────────┘
```

### 3.1 调度约束：工具自描述，不是系统猜测

调度相关的方法（`isReadOnly`、`isDestructive`、`isConcurrencySafe`）不含任何业务逻辑——它们是**纯粹的声明**。工具说"我是只读的"，调度器就可以放心并发执行多个同类工具；工具说"我会造成不可逆操作"，UI 就会在执行前加醒目警告。

**fail-closed 设计原则**：所有调度声明的默认值都是最保守的一侧。`isConcurrencySafe` 默认 `false`，意味着工具除非主动声明可以并发，否则强制串行。这是"假设最坏情况，直到被证明安全"的防御性设计，与权限系统的默认拒绝是同一哲学。

### 3.2 行为元数据：工具的"身份证"

除方法外，工具还携带一组元数据，让系统在不执行工具的情况下了解它：

- **`searchHint`**：3-10 个关键词，供 ToolSearch 索引——工具描述自己"在什么情境下有用"
- **`maxResultSizeChars`**：输出超过此阈值后自动持久化到磁盘，模型只看到摘要
- **`shouldDefer` / `alwaysLoad`**：控制工具 schema 是否加入初始 system prompt（详见第九节）
- **`isMcp` / `isLsp`**：来源标记，影响 UI 展示和权限策略

### 3.3 渲染层：工具控制自己的 UI 表达

工具可以实现多个可选渲染方法，为不同执行状态定制 UI：执行中的进度展示、被拒绝时的解释文字、出错时的帮助信息。这让工具的用户体验不依赖于通用的"工具调用框"，每个工具可以呈现最合适的交互形式。

> 参见 `src/Tool.ts`（完整接口定义）；`src/tools/BashTool/BashTool.tsx`（典型完整实现）

## 四、子模型调用：工具内嵌的模型编排

部分工具在执行过程中会发起自己的模型调用——工具不只是"执行动作"，它也可以"召唤另一个模型来思考"。

```
主对话模型（Opus/Sonnet）
    │
    ├─ WebSearchTool ──→ Haiku（处理搜索结果，提炼摘要）
    │
    ├─ AgentTool ──────→ 用户指定（haiku / sonnet / opus）
    │                     独立 token budget，独立上下文
    │
    └─ SkillTool ──────→ Skill 声明的模型
                          尊重 frontmatter 中的 model: 字段
```

| 工具 | 使用哪个模型 | 为什么 |
|------|------------|--------|
| `WebSearch` | Haiku（小型快速模型）| 原始搜索结果含大量冗余 HTML，用轻量模型过滤摘要，避免主模型消耗 token |
| `Agent` | 调用方指定 | 多代理编排的基础，不同子任务可以用不同能力/成本的模型 |
| `Skill` | frontmatter 声明 | 用户定义的工作流可以声明自己适合用哪个模型运行 |

**关键设计**：子模型调用通过 `QueryEngine` 的 fork 机制运行，拥有独立的 token budget 和消息上下文，执行结果不污染主对话的消息历史。这是工具系统支持真正"多模型编排"的底层保障。

> 参见 `src/tools/WebSearchTool/`、`src/tools/AgentTool/`、`src/tools/SkillTool/`；`src/QueryEngine.ts` 的 fork 机制

## 五、权限系统：多层防御，最小信任

工具权限系统的核心哲学是**最小信任原则**：每次工具执行都需要经过合法性验证，权限需要明确授予而不是默认拥有。

### 5.1 规则的来源与优先级

权限规则来自八个来源，优先级从高到低：

```
企业策略（policySettings）          ← 只读，管理员设置
    │
CLI 参数（flagSettings / cliArg）   ← 本次启动时传入
    │
项目配置（projectSettings）          ← .claude/settings.json
    │
本地配置（localSettings）           ← .claude/settings.local.json（不提交 git）
    │
用户配置（userSettings）            ← ~/.claude/settings.json
    │
当前 Skill 声明（command）          ← SKILL.md 的 allowed-tools
    │
会话临时规则（session）             ← 本次会话用户授权
```

高优先级来源的规则会覆盖低优先级的，企业策略始终具有最高权威。

### 5.2 规则格式：细粒度的行为匹配

权限规则使用工具名 + 可选的内容匹配模式，支持从粗到细的控制粒度：

```
Bash                    → 匹配所有 Bash 调用
Bash(git *)             → 只匹配以 git 开头的命令
Bash(npm run test)      → 精确匹配某条命令
Edit(src/*)             → 只匹配 src/ 下的文件编辑
mcp__my_server          → 封锁整个 MCP 服务器的所有工具
```

这套匹配语法让"允许 git 操作但禁止 rm"这类细粒度策略可以直接表达。

### 5.3 三路审批流水线

每次工具调用都经过三个阶段的审批，任意阶段通过即可执行：

```
工具调用请求
    │
    ▼ ① Hooks（预审批）
  settings.json 配置的 PermissionRequest hooks
  脚本返回 approve / deny / ask
  适用于：自动化审批策略、外部合规系统集成
    │
    ▼ ② Classifier（自动安全判断）
  仅当 BASH_CLASSIFIER feature 开启时生效
  将命令发给分类器模型判断风险级别
  低风险 → 自动批准，不打扰用户
  高风险 → 进入人工审批
    │
    ▼ ③ User Prompt（人工审批）
  弹出交互对话框
  用户选择：允许一次 / 始终允许（持久化）/ 拒绝
    │
    ▼
  执行工具 / 记录权限规则到 settings.json
```

**设计洞察**：Hooks 让企业可以在 Claude Code 外部挂载审批逻辑；Classifier 减少了用户面对低风险操作时的审批疲劳；User Prompt 是兜底的人工控制。三者各司其职，组合覆盖了从全自动到全手动的连续谱。

### 5.4 权限模式：会话级别的安全等级

不同场景需要不同的整体审批策略，权限模式（Permission Mode）提供了会话级别的安全等级切换：

```
bypassPermissions  ── 跳过所有检查（--dangerously-skip-permissions）
      │
    auto           ── 自动批准安全操作，高风险仍需人工
      │
   default         ── 每次工具调用都询问用户
      │
    plan           ── 规划模式，拦截所有工具执行
```

这不是权限的粗暴开关，而是对"自主性"和"控制权"之间权衡的显式管理。

### 5.5 拒绝跟踪：防止过度封锁

自动拒绝（auto-deny）达到阈值后，系统回退到人工提示模式。这解决了一个实际问题：过于激进的自动拒绝规则可能误伤合法操作，拒绝跟踪确保系统在连续拒绝后会停下来询问人。

> 参见 `src/hooks/toolPermission/PermissionContext.ts`（审批流水线）；`src/utils/permissions/permissions.ts`（规则匹配）；`src/types/permissions.ts`（权限类型定义）

## 六、MCP：打开工具系统的边界

MCP（Model Context Protocol）是工具系统对外的扩展接口。它的核心设计目标是：**让外部工具像内置工具一样被使用，而不是作为二等公民被特殊对待**。

### 6.1 统一接口的意义

```
内置工具（TypeScript）
    │                  ┌────────────────────────┐
    │                  │    MCPTool 适配器       │
MCP 外部工具 ─────────→│                        │→  Tool 统一接口
（任意语言/远程服务）  │  名称：mcp__服务器__工具  │
                       │  权限：默认 ask         │
                       └────────────────────────┘
```

MCPTool 是一个适配器，将外部服务器的工具能力映射到与内置工具完全相同的接口。调度器、权限系统、UI 渲染层无需区分工具来源——内置工具和 MCP 工具经过相同的管道处理。

命名约定 `mcp__服务器名__工具名` 起到命名空间隔离的作用：既避免与内置工具名称冲突，又让 deny rule 可以用 `mcp__server_name` 一次性封锁整个外部服务器。

### 6.2 连接方式

MCP 服务器支持多种传输协议，覆盖从本地进程到远程服务的全部场景：

| 类型 | 适用场景 |
|------|---------|
| `stdio` | 本地子进程（最常用，低延迟）|
| `sse` / `http` | 远程 HTTP 服务 |
| `ws` | WebSocket 实时连接 |
| `sdk` | Claude Agent SDK 集成 |
| `sse-ide` / `ws-ide` | IDE 扩展（VS Code / JetBrains）|
| `claudeai-proxy` | Claude.ai 云端代理 |

### 6.3 安全隔离

MCP 工具默认需要用户确认（`behavior: ask`），与内置工具的默认允许形成对比——外部工具来自第三方，不能预设信任。用户可以通过 allow rule 提升特定 MCP 工具的信任级别，或通过 deny rule 完全封锁某个服务器。

> 参见 `src/tools/MCPTool/`（适配器实现）；`src/services/mcp/`（服务器连接管理）

## 七、Tools vs Skills：执行原语 vs 意图工作流

Tools 和 Skills 是 Claude Code 中最容易混淆的概念对，但它们在系统中扮演截然不同的角色。

| 维度 | Tools（工具）| Skills（技能）|
|------|-------------|--------------|
| 本质 | 执行原语 | 可复用工作流 |
| 调用方 | 模型（通过 tool_use API）| 用户（通过 `/slash-command`）|
| 输入 | 结构化 JSON，Zod schema 验证 | 字符串参数 + markdown 模板 |
| 实现语言 | TypeScript（确定性逻辑）| Markdown（意图描述）|
| 执行者 | 工具本身 | 模型（接收 prompt 后调用 Tools 执行）|
| 扩展方式 | MCP 服务器 / 插件 | `SKILL.md` 文件 |
| 并发控制 | `isConcurrencySafe()` 声明 | `context: fork` 隔离 |

**一句话区分**：Skills 告诉模型"做什么"，Tools 是模型"怎么做"的手段。

### Skills 的执行链

```
用户输入 /my-skill 参数
    │
    ▼
SkillTool.call()  ← 找到对应的 SKILL.md
    │
    ▼ getPromptForCommand()
生成 markdown 指令 → 注入对话
    │
    ▼
模型接收 prompt + 当前工具池
    │
    ▼ 模型决策：调用若干工具
Read → Grep → Edit → Bash...
    │
    ▼
结果汇总，呈现给用户
```

### 两种执行上下文

**inline（默认）**：Skill prompt 注入当前对话，用户可以随时介入调整方向，执行过程在会话流中可见。

**fork**：通过 AgentTool 启动独立子代理，拥有独立 token budget，用户无法中途干预，适合需要隔离执行的批量任务。

> 参见 `src/skills/`（Skill 加载与注册）；`src/tools/SkillTool/`（执行逻辑）；`src/types/command.ts`（Command 类型定义）

## 八、用户如何引入新工具

Claude Code 提供三个层次的扩展能力，复杂度和灵活性依次递增：

```
Level 1：SKILL.md（markdown，无需编程）
    │  描述工作流意图，模型负责执行
    │  适合：复用会话模式、团队共享工作流
    ▼
Level 2：MCP 服务器（任意语言）
    │  实现工具逻辑，通过协议接入

    ▼

       提供 Skills + Tools + Hooks 的完整能力束
```

## 九、独特机制

### ToolSearch：按需加载工具 schema

将 45 个工具的完整 schema 全部注入 system prompt 会带来两个代价：**token 消耗**和 **prompt cache 失效**（工具列表变化会让整个 prompt 缓存失效）。

Claude Code 的解法是引入 `shouldDefer` / `alwaysLoad` 双标记：

```
alwaysLoad = true   → 高频核心工具，始终在 system prompt 中
shouldDefer = true  → 低频专用工具，初始不加载
（默认）            → 中频工具，按需决策
```

低频工具（如 `NotebookEdit`、`LSP`、`WebBrowser`）初始不出现在模型的工具列表中。当模型需要一个不认识的工具时，它调用 `ToolSearch("jupyter notebook")`，搜索器按 `searchHint` 关键词匹配，返回对应工具的完整 schema，模型这才"学会"用这个工具。

**本质上这是一种懒加载 + 语义检索的结合**：把工具发现的成本从"启动时全量加载"分摊到"需要时按需加载"，同时让模型通过自然语言描述找到工具，而不是依赖准确的工具名。

### 结果持久化：大输出的内存管理

工具输出超过 `maxResultSizeChars` 阈值时，不直接放入对话上下文，而是写入临时文件，只把摘要返回给模型。模型如果需要完整内容，可以用 `FileRead` 读取。

这解决了一个实际问题：`find /`、`git log --all`、大型测试报告等命令的输出可以达到数十万字符，直接放入 context 会挤压其他内容，甚至撑爆 context window。持久化机制让模型始终在"看摘要 + 按需展开"的模式下工作。

### Classifier：分类器驱动的自动审批

`BASH_CLASSIFIER` feature 开启后，Bash 命令在进入人工审批前会先经过一个分类器模型的判断。分类器的输入是命令字符串，输出是风险级别。

这是一个**信任分级**的设计：对明显安全的只读命令（`ls`、`cat`、`git status`）自动放行，只把真正需要人判断的操作推给用户。它在"不打扰用户"和"不放弃控制权"之间找到平衡点，而不是二选一。

工具通过实现 `toAutoClassifierInput()` 方法向分类器提供待判断的内容，默认实现返回空字符串（跳过分类器）——只有像 BashTool 这样的高风险工具才实现此方法，其他工具直接走人工审批通道。

### 并发安全：声明式并行调度

当模型在一轮中输出多个工具调用时，调度器会检查每个工具的 `isConcurrencySafe()` 返回值，将可并发的工具分为一批同时执行，不可并发的串行处理。

工具通过声明而非实现来参与并发决策——调度器不需要分析工具代码，只读取声明。`GrepTool`（只读搜索）声明可并发，多个 Grep 调用同时执行；`BashTool`（可能有副作用）声明不可并发，强制串行。

> 参见 `src/tools/ToolSearchTool/`（延迟加载实现）；`src/hooks/toolPermission/denialTracking.ts`（拒绝跟踪）；`src/query.ts` 的工具批次调度逻辑
