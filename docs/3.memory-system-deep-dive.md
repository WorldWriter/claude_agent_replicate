# 研究问题 3 - Memory：生成与召回的双轨架构

> 基于 2026-03-31 npm source map 泄露的源码快照进行分析。
> 核心文件：`src/memdir/`、`src/services/extractMemories/`、`src/services/autoDream/`、`src/utils/attachments.ts`

***

记忆系统是 Claude Code 跨会话"自我"的基础设施。每次对话结束后，它学到的东西如何留下来？下次对话开始时，它又如何找回有用的知识？这篇文章从两个维度拆解这个问题：**记忆如何被写入**（四条生成通道）以及**记忆如何被读取**（三层默认召回 + 可选的智能预取）。

存储模型本身很简单：每条记忆是一个带 frontmatter 的 Markdown 文件，`MEMORY.md` 作为索引（目录而非内容）。复杂性全部集中在"何时写、谁来写、写什么"以及"何时读、读多少、如何不阻塞主流程"这两个问题上。

***

## 一、记忆的生成——四条并行通道

### 1.1 用户主导：显式学习（Human-in-the-loop）

`/memory` 命令是记忆系统唯一的人工入口。执行后，它打开一个交互 UI，让用户从现有记忆文件中选择一个或新建，然后在 `$VISUAL` 或 `$EDITOR` 中打开编辑。

这条通道的频率最低，但权威性最高——它是人类直接校准 AI 行为的地方。系统提示中关于"何时应该忘记某事"的指导（"如果用户要求你忘记，删除相关条目"）也指向这条通道。

用户也可以不通过 `/memory` 命令，而是在对话中直接说"帮我记住这件事"——这实际上触发的是下一条通道（主 Agent 机会式写入），因为系统提示中明确将其列为触发条件之一。

***

### 1.2 主 Agent 机会式写入（Opportunistic Learning）

**本质：普通 Tool Call，不是新 Agent**

这是一个重要的架构澄清：主 Agent 写记忆文件，走的是与写任何其他文件完全相同的代码路径，没有特殊分支，没有 fork 新 session。区别仅在于 system prompt 中注入了"操作手册"——`buildMemoryLines()` 把写入规范嵌入系统提示，模型读到手册后，**自主决定**何时写、写什么，然后在当前对话 loop 里直接调用 Write tool 完成。

写入规范要求两步操作：

1. 先写主题文件（含标准 frontmatter：name / description / type），
2. 再在 `MEMORY.md` 中追加一行索引（单行，≤150 字符）。

这两步是业务规范层面的"原子性"（而非代码事务），确保索引与内容始终同步。

**与守护 Agent 的互斥**：主 Agent 写入后，守护 Agent 通过扫描消息历史中的 Write/Edit 工具调用痕迹来检测这一事实，然后跳过本轮提取。互斥不靠锁，靠的是"已有记录即真相"——消息历史本身就是协调信号。

***

### 1.3 后台守护 Agent：事后提取（Post-Query Guard）

这是记忆系统的"安全网"。当主 Agent 专注于任务、来不及或不认为某条信息值得记忆时，守护 Agent 会在事后回顾并补录。

**触发时机**：每次完整对话回合结束后由 `handleStopHooks()` 以 fire-and-forget（`void`）方式启动，不阻塞用户继续输入。完成后在 UI 中注入一条简短的"记忆已保存"系统消息，事后告知而非阻塞式进度条。

**实现模式：完美 Fork + 受限权限**

守护 Agent 是主对话的完美 fork——继承父会话的 prompt cache，无需重新付出 cache creation 的 token 开销。但它的工具权限被严格裁剪：读取类工具不受限；写入类工具只允许操作 memory 目录，任何离开该目录的写操作都会被拦截；删除和任意 Bash 写操作被完全禁止。

**执行效率设计**：maxTurns=5 硬上限，内部策略是"第 1 轮并行读取、第 2 轮并行写入"，2 轮即可完成核心工作。

**合并执行（Coalesce）**：如果上一次提取尚未完成，新触发的上下文不会开启新的 Agent，而是被 stash，等当前提取完成后立即以追加形式运行。这避免了并发写入同一记忆文件的竞争条件。

**优雅退出**：进程退出前最多等 60 秒，确保飞行中的提取任务完成，知识不因进程退出而丢失。

***

### 1.4 长周期蒸馏：autoDream（Nightly Consolidation）

前三条通道都是"实时"的——在单次对话的生命周期内发生。`autoDream` 是唯一一条跨越多个历史会话的"离线"通道，负责把散落在多个 session transcript 中的零碎事实系统性地整理进记忆。它与 KAIROS 模式（实时日记功能）**互斥**：两者解决同一问题但信息源不同，KAIROS 开启时 autoDream 直接退出。

**触发时机**：同守护 Agent 一样，每次对话回合结束后由 `handleStopHooks()` fire-and-forget 启动，后台静默运行，不阻塞用户。但实际上**每次都触发检查、偶尔才真正执行**——五重门控让绝大多数回合在时间门（≥24h）处直接 return。完成后追加 inline system message 汇报修改摘要。

#### 触发：五重门控

每次对话结束后依次检查，全部通过才启动：

```
① 功能开关     内存判断，最廉价
② 时间门       距上次整理 ≥ 24 小时
③ 扫描节流     10 分钟内不重复 stat
④ 会话门       自上次整理后 ≥ 5 个新 session（不含当前）
⑤ 文件锁       mtime-based 文件锁，最昂贵，防并发
```

顺序不是随意的：最廉价的判断排前面，前面的门没过不执行后面更贵的操作。

#### 文件锁：一个文件，三重职责

锁文件（`.consolidation-lock`）内存 PID，mtime 同时充当"上次整理时间戳"：

| 职责   | 机制                                      |
| ---- | --------------------------------------- |
| 互斥锁  | 检查 PID 对应进程是否存活，60 秒内有效                 |
| 时间记录 | mtime = lastConsolidatedAt，时间门直接读 mtime |
| 崩溃自愈 | 进程崩溃后 PID 失效，下次自动抢锁                     |

**失败回滚**：整理 Agent 抛出异常时，mtime 倒回整理开始前——时间门下次依然通过，不因一次失败推迟 24 小时。

#### 执行：四阶段整理

| 阶段                          | 目标                                             | 关键约束                                                   |
| --------------------------- | ---------------------------------------------- | ------------------------------------------------------ |
| **Phase 1 — Orient**        | `ls memory/`，读 MEMORY.md，浏览现有主题文件              | 先建全貌心智模型，避免后续创建重复                                      |
| **Phase 2 — Gather**        | 采集新信号：日记日志 → 已漂移的旧记忆 → transcript 搜索           | **不全量读 transcript**，只在知道找什么时才 grep，防大 JSONL 撑爆 context |
| **Phase 3 — Consolidate**   | 合并进已有主题文件，相对时间转绝对日期，删除被否定的旧记录                  | 合并而非新建，保持记忆不冗余                                         |
| **Phase 4 — Prune & Index** | 保持 MEMORY.md ≤ 200 行 / 25KB，每行 ≤150 字符，解决文件间矛盾 | 索引行过长说明内容混入了索引，细节应移到主题文件                               |

整理完成后，如有文件被修改，主 transcript 追加 inline system message 汇报摘要，对用户半透明可见。

#### 两个工程细节

**skipTranscript**：autoDream fork 的对话不记录进 session transcript——否则下次整理时自己上次的 tool calls 会被当成对话内容分析，造成自我引用噪音。

**进度状态机**：任务有两个阶段，`starting`（读取分析中）→ `updating`（第一个 Write/Edit 落地时翻转）。UI 实时追踪文本推理、工具调用数量、已触碰的文件路径。

## 二、记忆的召回——静态注入与动态 Attachment

生成是写，召回是读。Claude Code 的记忆召回不是"查数据库"，而是将记忆内容直接注入进上下文窗口。注入分两条主轨：**静态召回**（对话启动时一次性固定注入）和**动态召回**（对话进行中由事件驱动注入，以 Attachment 形式实现）。

***

### 2.1 静态召回：会话启动时固定注入

静态召回在每次对话启动时执行一次，结果被 prompt cache 缓存，多轮共享。它构成 Claude 记忆感知的"基线"。

**注入内容一：MEMORY.md 索引**（`src/memdir/memdir.ts`）

`loadMemoryPrompt()` 把 MEMORY.md 的完整内容注入 system prompt，同时注入写入规范（两步写入指引）。这让 Claude 在对话伊始就知道"自己记住了哪些事、对应哪些文件、如何正确写入新记忆"。

三种模式分发：

```
KAIROS 模式  → 注入日记日志路径 pattern（跨日期 cache 稳定）
Team 模式    → 注入双份索引（私人 + 团队共享 MEMORY.md）
默认模式     → 注入个人 MEMORY.md 内容 + 写入规范
```

Memory section 位于 system prompt 的动态边界（`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`）之后，属于用户/项目特定区域，不污染全局 cache。MEMORY.md 内容超限时双重截断：先按行（200行），再按字节（25KB），超限后追加警告 banner。

**注入内容二：CLAUDE.md 文件**（`src/context.ts`）

`getUserContext()` 扫描项目层级中的 CLAUDE.md、CLAUDE.local.md 和 `.claude/rules/*.md`（过滤 memory 目录），注入 user context。这是另一套平行通道——用户手写的配置型记忆，永远在上下文里。与 auto memory 的区别：CLAUDE.md 是人工维护的规则，auto memory 是 AI 自主生成的经验。

***

### 2.2 动态召回：Attachment 驱动的两种机制

动态召回通过 Attachment 系统实现，在对话进行中由特定事件触发，将记忆内容以 `system-reminder attachment` 形式注入，绕过 per-message tool-result 的 token 预算。源码中定义了三种记忆相关的 attachment 类型，但实际运作的只有两种：

#### 机制一：nested\_memory（默认开启，无 feature flag）

**触发条件**：每次 FileReadTool 读取文件时自动触发，无需任何开关。触发点在工具执行层，不区分调用动机——来自用户指令、LLM 主动推理、还是 MEMORY.md 索引引导，效果完全相同。

**三类触发来源**：

```
① 用户主动："帮我看一下 src/api/client.ts"
  → LLM 调用 FileReadTool → 触发 nested_memory for src/api/

② LLM 推理主动：判断需要读某文件来回答问题
  → LLM 调用 FileReadTool → 触发 nested_memory

③ MEMORY.md 索引引导：LLM 在索引里看到某条记忆，主动去读它
  → LLM 调用 FileReadTool("~/.claude/memory/user_role.md")
  → 触发 nested_memory for ~/.claude/memory/
```

第③条产生一个联动效应：静态召回里的"MEMORY.md 索引"作为入口，驱动 LLM 主动读取记忆文件，该读取又作为副效应触发 nested\_memory，自动补充记忆目录的上下文规则。

**工作原理**：读取一个文件时，系统会沿目录树（从 CWD 到目标文件所在目录）逐层查找 CLAUDE.md、unconditional rules、conditional rules，将沿途发现的规则文件注入。本质上是"读哪个目录的文件，就自动补充那个目录的上下文规则"。

**去重机制**：通过 session 级别的永久 Set 追踪已注入过的文件，同一文件在整个会话内只注入一次。

这个机制的意义在于：它让目录级别的上下文感知变得自动化——当 Claude 开始操作某个模块的代码时，该模块的 CLAUDE.md 规则自然地进入上下文，无需用户手动引导。

#### 机制二：relevant\_memories（默认关闭，需开启 tengu\_moth\_copse）

**触发条件**：每轮对话开始时，由 feature flag `tengu_moth_copse` 控制。默认关闭。

**工作原理**：启动一个与主对话完全并行的 Sonnet sideQuery，语义匹配当前对话最相关的记忆文件，在工具调用结束后注入：

```
轮次开始
  ├── 启动异步预取（立即返回 handle，不阻塞）
  │     ↓ 后台 Sonnet sideQuery
  │   输入：用户问题 + 所有记忆的 frontmatter 元数据
  │         + 最近成功调用的工具列表 + 已注入路径（去重）
  │   输出：选中的 ≤5 条记忆文件 → 读取内容
  │
  ├── 主 Agent 流式输出 + 工具调用（并行进行）
  │
  └── 工具调用结束后
        ├── 预取已结算 → 注入本轮
        └── 预取未结算 → 下轮重试（eventual consistency，绝不阻塞）
```

**Noise suppression**：对于最近已成功使用的工具，其参考文档类记忆不选（模型已在用了）；但该工具的警告/gotcha 类记忆仍然选择。

**预算管控**：

| 级别   | 限制                                               |
| ---- | ------------------------------------------------ |
| 单个文件 | 4KB / 200 行（超出追加"Use FileReadTool to view full"） |
| 单轮注入 | ≤5 条 × 4KB = 20KB                                |
| 会话累计 | 60KB（超出后停止预取）                                    |

每条记忆携带新鲜度 header（`saved 3 days ago` / `STALE (30+ days)`），帮助 Claude 判断信息可信度。资源释放通过 `using` 语句绑定 dispose，所有退出路径均正确 abort 飞行中的 sideQuery。

#### 第三种类型：current\_session\_memory（仅类型定义，未实装）

源码中定义了 `current_session_memory` 这个 attachment 类型，但整个代码库中没有任何地方创建它——只是一个为未来预留的占位符，当前不起任何作用。

***

### 两种动态机制的对比

| <br />   | nested\_memory        | relevant\_memories      |
| -------- | --------------------- | ----------------------- |
| **默认状态** | 开启                    | 关闭（tengu\_moth\_copse）  |
| **触发方式** | FileReadTool 读文件时自动触发 | 每轮对话开始时主动触发             |
| **选择逻辑** | 目录树遍历，规则自动关联          | Sonnet 语义匹配，最相关优先       |
| **注入内容** | CLAUDE.md / 目录规则文件    | auto memory 目录中的记忆文件    |
| **去重**   | session 级 Set，永不重复注入  | LRU + session 路径集合      |
| **成本**   | 零额外 API 调用            | 一次 Sonnet sideQuery / 轮 |

## 三、设计哲学与关键权衡

### 写入侧：用消息历史做协调，不用锁

四条生成通道通过"主 Agent 写入 → 守护 Agent 跳过"协作。互斥不靠共享可变状态，而是靠扫描消息历史中已有的 tool call 记录——消息历史本身就是协调信号，读取不需要任何同步原语。

### 召回侧：三层兜底 + 一层增强

默认的三层（index in prompt → CLAUDE.md → 模型主动读）形成一个完整的兜底链，无需任何额外功能开关。智能预取是锦上添花——它把"模型主动判断是否需要读"这一认知负担转移到一个专门的选择器，让主 Agent 更专注于任务本身。

### autoDream：离线蒸馏的工程约束

autoDream 最核心的工程约束是"不要全量读 transcript"——这条约束由提示词而非代码强制。这体现了一种设计哲学：对于 Agent 行为的约束，有时候写在提示词里比写在代码里更灵活，但也意味着它依赖模型的执行忠实度。

### Cache 友好性贯穿始终

整个系统的设计都在最大化 prompt cache 利用率：

- 静态注入的 memory section 缓存在 session 级别
- 守护 Agent 和 autoDream fork 继承父会话 cache
- KAIROS 模式下日志路径用 pattern 而非字面日期，确保跨午夜的 cache 不失效
- autoDream 文件锁的 mtime 设计让"时间记录"和"互斥锁"合二为一，减少一次文件读取

### 多重预算防止上下文溢出

记忆系统的最大风险是"把上下文窗口塞满记忆"。系统在每个层面都设置了硬截断：文件级（4KB）、每轮（20KB）、会话（60KB）、索引文件（25KB）。截断时留下明确提示，告知 Claude 内容已被截断以及如何获取完整内容。

### 存储格式示例

**记忆主题文件**（如 `~/.claude/projects/<slug>/memory/user_prefs.md`）：

```markdown
---
name: 用户偏好
description: 用户的代码风格和工作习惯偏好
type: feedback
---

不要在函数末尾加总结性注释，用户认为多余。

**Why:** 用户明确说过"代码能说明自己"。
**How to apply:** 写代码时不加 # end of function 类注释。
```

`type` 四种取值及其写入范围：

| type | 含义 | 写入范围 | 存储路径 |
| --- | --- | --- | --- |
| `user` | 用户的角色、目标、知识背景 | 永远 **private** | `memory/` |
| `feedback` | 用户对 AI 行为的偏好和纠正 | 默认 private；仅当是项目级约定（测试策略、构建规范）时才写 team | `memory/` 或 `memory/team/` |
| `project` | 项目进展、决策、背景（代码库之外的） | 强烈倾向 **team** | `memory/team/` |
| `reference` | 外部系统指针（Linear、Grafana、Slack 等） | 通常 **team** | `memory/team/` |

`private` 路径为 `~/.claude/projects/<slug>/memory/`，`team` 路径为其子目录 `team/`。scope 判断**完全依赖提示词指导**，由模型自主决定写哪个目录，无代码强制。Team Memory 模式需 feature flag `tengu_herring_clock` 开启，默认关闭。

---

**MEMORY.md 索引文件**（`~/.claude/projects/<slug>/memory/MEMORY.md`）：
