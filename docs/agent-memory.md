# Claude Agent Memory 研究（设计视角）

## 1. 核心问题解答

### Q1: 哪些记忆会被加载到 system prompt？对话过程中是否还可以召回记忆？

- **初始化加载 (静态轨)**：在每次对话开始时，记忆系统的“元规则”和“索引”会被 **静态注入** 到 System Prompt 中。具体来说，`MEMORY.md` 文件的内容和关于如何读写记忆的规范（来自 `src/memdir/memdir.ts`）会作为初始上下文的一部分。这确保了 Claude 在与你对话前，就已经知道了“哪些事值得记”、“如何更新记忆”以及“何时应该忽略记忆”。

- **对话中召回 (动态轨)**：在对话过程中，记忆是 **动态召回** 的。这通过一个名为 `startRelevantMemoryPrefetch` 的异步预取机制实现（见 `src/utils/attachments.ts`）。它会根据你“当前的问题”和 Claude“最近的行动”，在后台并行地检索最相关的记忆文件。因为是并行预取，这个过程几乎不会影响主对话的响应速度，实现了工具执行与知识召回的无缝衔接。

### Q2: 用户主动编辑的记忆和 Claude 自动运行的记忆都是怎么产生的？

Claude 的记忆生成是一个“人机协作”的混合模型，主要有四种方式：

1.  **用户主导 (显式学习)**：你可以通过 `/memory` 命令或直接要求 Claude “记住/忘记”某事来直接管理记忆。这是最直接的“人类校准”方式。
2.  **主 Agent 决策 (机会式学习)**：在对话中，如果 Claude 自主判断某条信息（比如一个重要的项目决策或你的个人偏好）对于未来的长期合作有价值，它会直接使用 `Write` 或 `Edit` 工具将信息写入记忆目录。
3.  **后台 Agent 整理 (守护式学习)**：在每次完整的对话回合结束后，系统会派生一个轻量级的“守护 Agent”（见 `src/services/extractMemories/extractMemories.ts`）。这个 Agent 的权限非常受限，它会回顾刚刚的对话，以“自监督”的方式将可能被遗漏的重要信息整理并存入记忆，实现了“主线程专注任务，后台线程整理知识”的分工。
4.  **长周期蒸馏 (夜间巩固)**：当 Claude 处于空闲状态时，一个名为 `autoDream` 的服务（见 `src/services/autoDream/autoDream.ts`）会启动。它会巡检多个历史会话的日志，将其中有价值但零散的事实提炼、归档到结构化的记忆文件中，形成一种“日常学习 + 夜间巩固”的节奏。

### Q3: 记忆以什么形式存储？是否有上限？相关性如何召回？

- **存储形式**：记忆并非一个巨大的文件，而是一个 **“索引 + 文件”** 的系统。
    - **索引**：`MEMORY.md` 是记忆的“目录”，每行只包含一条记忆的简短描述和指向具体文件的链接。
    - **正文**：每条具体的记忆都是一个独立的 Markdown 文件，其头部通过 `frontmatter` 格式定义了元数据（如 `name`, `description`, `type`）。这种设计将“目录”与“内容”分离，便于快速检索和按需加载。

- **存储上限**：为了防止上下文窗口被撑爆，记忆系统有严格的预算治理：
    - **单轮注入上限**：最多注入 5 条记忆，每条不超过 4KB。
    - **会话累计上限**：整个会话中，从记忆系统加载的总内容不超过 60KB。
    - **索引文件上限**：`MEMORY.md` 文件本身也被限制在 200 行或 25KB 以内，强制要求将细节写入独立文件。

- **相关性召回**：Claude **不使用传统的向量数据库** 进行召回。它采用了一种更轻量且更智能的“元信息选择器”：
    - 它会分析记忆文件的元数据（文件名、描述、类型）和你的问题，来判断相关性。
    - 这种方法的好处是无需外部索引服务，并且可以结合更多上下文信息。例如，如果 Claude 发现你正在频繁使用某个工具，它可能会降低注入该工具相关 API 文档记忆的优先级，因为你很可能已经熟悉它了，从而最大化利用宝贵的上下文空间。

### Q4: 其他研究报告里的内容

报告中还提到了记忆系统与其他几个关键子系统的联动，这有助于我们理解其在整个架构中的位置：

- **与上下文压缩的关系**：即使旧的对话历史被压缩或遗忘，只要记忆文件还在磁盘上，相关信息就依然可以在未来的对话中被重新召回，形成“短期对话 vs. 长期记忆”的互补。
- **与工具权限的耦合**：所有自动化的记忆写入操作都受到严格的工具权限控制。守护 Agent 只能在指定的记忆目录中写入，任何“越界”操作都会被拒绝，确保了记忆系统不会成为安全漏洞。
- **与多 Agent 协作的关联**：当派生子 Agent 时，它们会继承父 Agent 的记忆框架，但可以通过 `@agent-*` 这样的命名空间来限定自己的记忆目录，从而实现“每个 Agent 拥有独立记忆空间”的隔离设计。

## 2. 原始研究笔记

### 设计目标
- **将记忆视为第一类系统资源**：Claude Code 把 `MEMORY.md` 与 CLAUDE.md 等指令文件放在同一加载链中（`src/utils/claudemd.ts`），确保“跨会话自我”在对话启动前就具备稳定的行为约束与自省提示。
- **记忆生命周期标准化**：`src/memdir/memdir.ts` 中的系统提示定义了“写入→索引→检索→注入”的完整闭环，与研究计划提出的生命周期完全对齐（`docs/research-plan.md`）。
- **面向多场景的弹性策略**：通过 feature flag 组合（auto memory、team memory、Kairos 日志模式），同一框架可以覆盖个人项目、团队共享以及长驻助手等差异化需求。

### System Prompt 注入与运行期召回
- **分层注入思路**：system prompt 分为缓存前缀、缓存后段以及每轮变动段。记忆说明位于缓存段，保证所有回合共享相同的“记忆准则”，同时降低 token 开销（`src/constants/prompts.ts`）。
- **“静态 + 动态”双轨上下文**：
  - 静态轨：`MEMORY.md` 的内容与写入规范一起被注入，使模型在没有任何用户输入时就知道“哪些事情值得记住、如何更新索引、何时应忽略记忆”。
  - 动态轨：`src/utils/attachments.ts` 中的 `startRelevantMemoryPrefetch` 实现异步侧查询，将“当前问题 + 最近行动”映射到最相关的记忆文件，形成工具执行与知识召回的流水线。prefetch 的存在意味着记忆召回与主回合并行，对响应延迟几乎零影响。
- **上下文预算治理**：限制单轮注入数量、尺寸（5 条 / 每条 4KB）以及会话累计 60KB，配合 `truncateEntrypointContent()` 的 200 行 / 25KB 机制，保证记忆永远不会压垮 prompt。

### 记忆生成的角色划分
- **用户主导的“显式学习”**：任何用户主动提出“记住/忘记”的请求都会被系统提示强化，CLI 也提供 `/memory` 命令帮助用户直接操作记忆文件，形成“人类校准”入口。
- **主 Agent 的“机会式学习”**：如果 Claude 自己认为某条信息值得跨会话保留，可以直接使用 Write/Edit 工具写入 memory 目录。系统会检测到这种行为并跳过后续的自动提取，避免重复劳动。
- **后台 Agent 的“守护式学习”**：`src/services/extractMemories/extractMemories.ts` 针对每次完整回合结束都会 fork 一个轻量子 agent。它只访问最近几条对话、只拥有读取/写入记忆目录的权限，以“自监督”方式把遗漏的信息补齐。这个设计体现了“主线程专注任务、后台线程负责知识整理”的分工理念。
- **长周期蒸馏**：`src/services/autoDream/autoDream.ts` 会在空闲时对多 session 进行巡检，把散落在日志里的事实归档到记忆文件，形成“日常学习 + 夜间巩固”的节奏。

### 存储模型与检索策略
- **文件即知识单元**：每条记忆都是一个独立 Markdown，frontmatter 明确 `name`、`description`、`type`。四类记忆（user/feedback/project/reference）把“行为准则”“用户画像”“项目外部脉络”区分开，避免被代码中可推导的信息污染。
- **索引 vs. 正文**：`MEMORY.md` 被视作“目录而非内容”，其严格的一行一条策略迫使模型把细节写入主题文件，通过链接引用。这样的分离既方便检索，也让系统可以只加载目录（快速 orientation）而把正文留给按需召回。
- **相关性召回**：Sonnet 选择器使用文件元信息和自然语言描述决定是否值得注入，而不是直接使用嵌入向量库。好处是：1）无需额外索引服务；2）可根据最近工具使用情况进行语义抑制（如工具正在被频繁调用时不再注入 API 文档）；3）结合 `readFileState`，能感知模型已经看过哪些文件，最大化上下文利用率。

### 与其他子系统的耦合
- **上下文压缩**：当旧对话被压缩或剪裁时，只要记忆文件还在磁盘，prefetch 仍可重新注入，因此记忆系统与 `src/services/compact` 形成“短期对话 vs. 长期记忆”的互补。
- **工具权限模型**：自动写入完全依赖受限工具集，并通过 `createAutoMemCanUseTool` 控制作用域。任何离开记忆目录的写操作都会被拒绝，确保“持久化层”不会成为越权通道。
- **多 Agent 协作**：子 agent 运行时会继承同样的记忆框架，但可以通过 `@agent-*` 前缀来限定召回目录，实现“每个 agent 有独立记忆空间”的隔离设计。

### 研究延伸
- 学完记忆系统后，可按研究计划建议回到上下文压缩、工具权限和多 agent 管线，观察记忆如何与计划/任务、权限申请、协作沟通联动。
- 另一个值得跟进的方向是 Kairos 模式：它把“实时索引”替换为“日记 + 夜间 distill”，体现了 Claude 在长会话场景下对稳定性与可操作性的再权衡。
